    17. 1. DB Life Cycle

    The Database Life Cycle (DBLC) is a structured approach to designing, implementing, and maintaining a database system. It consists of several phases:

    1. Planning: Define the purpose, scope, and objectives of the database.
    2. Requirements Analysis: Gather and analyze user requirements.
    3. Conceptual Design: Create an Entity-Relationship (ER) model.
    4. Logical Design: Transform the ER model into relational schemas.
    5. Physical Design: Define physical storage structures and access methods.
    6. Implementation: Create the database and applications.
    7. Testing and Deployment: Test the system and deploy it.
    8. Maintenance: Monitor, backup, and update the database.

    2. File Based System & its Disadvantages and Limitations

    A file-based system stores data in separate files, each managed by individual applications. While simple, it has significant drawbacks:

    Disadvantages

    • Data Redundancy: Same data stored in multiple files, leading to inconsistency.
    • Data Inconsistency: Updates in one file may not reflect in others.
    • Data Isolation: Difficult to access data from multiple files simultaneously.
    • Data Integrity Issues: No centralized control over data validation.
    • Security Problems: Limited access control and encryption.
    • Concurrency Issues: Multiple users accessing files can cause conflicts.
    • Scalability Limitations: Hard to handle large volumes of data.
    • Backup and Recovery: Manual and error-prone.

    Limitations

    • No support for complex queries or relationships.
    • Dependency on application programs for data management.
    • High maintenance costs due to duplication.

    3. DBMS Advantages & Disadvantages

    A Database Management System (DBMS) is software that manages databases, providing an interface between users and data.

    Advantages

    • Data Independence: Applications don't need to know physical storage details.
    • Data Integrity: Ensures data accuracy and consistency through constraints.
    • Data Security: Access controls and encryption.
    • Data Sharing: Multiple users can access data concurrently.
    • Data Redundancy Control: Centralized storage reduces duplication.
    • Backup and Recovery: Automated mechanisms.
    • Query Language: SQL for easy data manipulation.
    • Scalability: Handles large datasets efficiently.

    Disadvantages

    • Cost: High initial and maintenance costs.
    • Complexity: Requires skilled personnel.
    • Performance Overhead: Additional processing for management tasks.
    • Vendor Lock-in: Dependency on specific DBMS.
    • Data Conversion: Migrating from file-based systems is challenging.

    4. ERD Notations

    Entity-Relationship Diagrams (ERDs) use specific notations to represent entities, attributes, and relationships.

    Basic Notations

    Element Notation Description
    Entity Rectangle Represents a real-world object or concept
    Attribute Oval Property of an entity
    Relationship Diamond Association between entities
    Key Attribute Oval with underline Unique identifier
    Multivalued Attribute Double oval Can have multiple values
    Derived Attribute Dashed oval Calculated from other attributes
    Weak Entity Double rectangle Depends on another entity for identification

    5. Entities & Attributes & Relations

    Entities

    An entity is a real-world object or concept that can be distinctly identified. Examples: Student, Course, Employee.

    Attributes

    Attributes are properties that describe an entity. Types:

    • Simple: Atomic values (e.g., Age)
    • Composite: Made of sub-parts (e.g., Address: Street, City, Zip)
    • Single-valued: One value per entity
    • Multivalued: Multiple values (e.g., Phone Numbers)
    • Derived: Calculated (e.g., Age from Birth Date)

    Relationships

    Relationships show how entities are related. They have:

    • Name: Describes the relationship
    • Degree: Number of participating entities
    • Cardinality: Number of instances
    • Participation: Whether participation is mandatory or optional

    6. Degree of Relation

    The degree of a relationship is the number of entity types participating in it.

    • Unary (Degree 1): Relationship within the same entity (e.g., Employee manages Employee)
    • Binary (Degree 2): Between two entities (e.g., Student enrolls in Course)
    • Ternary (Degree 3): Between three entities (e.g., Supplier supplies Part to Project)
    • N-ary: More than three entities

    7. Cardinality

    Cardinality specifies the number of instances of one entity that can be associated with instances of another entity.

    Type Description Example
    One-to-One (1:1) One instance of A relates to one instance of B Person has one Passport
    One-to-Many (1:N) One instance of A relates to many instances of B Department has many Employees
    Many-to-One (N:1) Many instances of A relate to one instance of B Many Students enroll in one Course
    Many-to-Many (M:N) Many instances of A relate to many instances of B Students enroll in many Courses

    8. Participation

    Participation indicates whether all instances of an entity must participate in a relationship.

    • Total Participation: Every instance must participate (double line in ERD)
    • Partial Participation: Some instances may not participate (single line)

    9. Keys & Constraints

    Keys

    Key Type Description
    Super Key Set of attributes that uniquely identify an entity
    Candidate Key Minimal super key (no subset is a super key)
    Primary Key Chosen candidate key for unique identification
    Foreign Key Attribute that references primary key of another table
    Alternate Key Candidate keys not chosen as primary
    Composite Key Primary key consisting of multiple attributes

    Constraints

    • Entity Integrity: Primary key cannot be null
    • Referential Integrity: Foreign key must reference existing primary key
    • Domain Constraints: Attribute values must be from valid domain
    • Key Constraints: Unique values for keys

    10. Mapping of Regular Entity Types

    Regular entities map directly to tables. Each entity becomes a relation with attributes as columns.

    • Entity: EMPLOYEE (EmpID, Name, Salary)
    • Table: EMPLOYEE(EmpID PK, Name, Salary)

    11. Mapping of Weak Entity Types

    Weak entities depend on strong entities. They include the primary key of the owner entity.

    • Strong Entity: DEPARTMENT (DeptID PK, Name)
    • Weak Entity: EMPLOYEE (EmpID, Name) owned by DEPARTMENT
    • Table: EMPLOYEE(DeptID FK, EmpID, Name) with composite PK (DeptID, EmpID)

    12. Mapping of Binary 1:1 Relation Types

    For 1:1 relationships, merge into one table or use foreign keys.

    • Option 1: Merge tables if total participation
    • Option 2: Add FK in one table, unique constraint

    13. Mapping of Binary 1:N Relationship Types

    Add primary key of 1-side as foreign key in N-side table.

    • DEPARTMENT (DeptID PK)
    • EMPLOYEE (EmpID PK, DeptID FK)

    14. Mapping of Binary M:N Relationship Types

    Create a new table with primary keys of both entities as composite primary key.

    • STUDENT (StudentID PK)
    • COURSE (CourseID PK)
    • ENROLL (StudentID FK, CourseID FK, Grade) with PK (StudentID, CourseID)

    15. Mapping of N-ary Relationship Types

    For relationships with degree > 2, create a new table with primary keys of all participating entities.

    • SUPPLIER (SupplierID PK)
    • PART (PartID PK)
    • PROJECT (ProjectID PK)
    • SUPPLY (SupplierID FK, PartID FK, ProjectID FK, Quantity) with PK (SupplierID, PartID, ProjectID)

    16. Mapping of Unary Relationship

    For recursive relationships, add a foreign key referencing the same table.

    • EMPLOYEE (EmpID PK, ManagerID FK) where ManagerID references EmpID
